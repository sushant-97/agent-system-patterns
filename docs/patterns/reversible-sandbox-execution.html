Reversible Sandbox Execution

Explore in a sandbox. Promote only safe diffs.

This pattern places risky agent work behind snapshots, isolated attempts, verification, and merge policy. The canonical workspace stays protected until a candidate diff proves it is safe enough to promote.

Core claim

Autonomous agents need exploration room, but product systems need recovery, auditability, and explicit promotion gates.

Controller questions

  • What state is the agent allowed to touch?
  • Can this attempt be fully rolled back?
  • Which changes are safe to promote?

Best use cases

  • Stateful edits to files, data, or dependencies.
  • High-blast-radius work needing auditability.
  • Cases where multiple candidate attempts should be compared.

Implementation shape

The Python model includes immutable checkpoints, isolated sandbox attempts, invariant verification, diffing, and a merge policy that can reject risky state transitions.

Logic

Promotion mechanics

1. Create checkpoint

The controller snapshots canonical state before the agent begins so every attempt has a stable fork point.

2. Run in isolation

The agent edits a sandbox copy instead of the durable workspace, which keeps failed attempts inspectable and reversible.

3. Verify and diff

Invariant checks and explicit change records tell the controller exactly what changed and whether the result is acceptable.

4. Promote selectively

Only changes that satisfy verification and merge policy are applied back to canonical state.

Verification

Experiment signals

Safe UI copy

A limited file edit passes verification and is promoted back to canonical state.

Destructive billing migration

The merge policy rejects state changes that would delete or damage durable data.

External side effect

Attempts that create non-reversible external effects are blocked even if the local diff looks harmless.

Competing attempts

Multiple sandboxes can be compared so the system promotes the safer or more correct candidate.
