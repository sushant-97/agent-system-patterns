Paged Attention: A Detailed Hands-On Guide

Agent System Patterns · Deep read

SP
Sushant Pargaonkar June 2026 · 14 min read

Serving LLMs efficiently isn't just about fast matrix multiplications; it's heavily bottlenecked by memory management. The KV cache (which stores previous keys and values for attention) grows dynamically as a model generates tokens. Until recently, LLM serving engines reserved large contiguous chunks of memory per request, leading to massive fragmentation. PagedAttention, pioneered by the vLLM project, applies an operating system concept—virtual memory and page tables—to solve this.

In this post, we will walk through exactly how PagedAttention works under the hood with detailed hands-on examples, animated data flows, and Python pseudocode to model the memory allocator.

The Trap of Contiguous Allocation

When a user sends a prompt to an LLM, the system cannot predict exactly how many tokens will be generated. The sequence could stop after 10 tokens, or continue for 2,000.

Before PagedAttention, systems like FasterTransformer would guess the maximum possible length (e.g., 2048 tokens) and allocate a contiguous block of VRAM for the KV cache. This leads to two critical problems:

  • Internal Fragmentation: If the model stops at 100 tokens, the remaining 1948 tokens of memory are locked up and wasted. This historically resulted in 60-80% wasted memory.
  • External Fragmentation: The available memory becomes scattered in small holes, meaning a new large request cannot be processed even if total free memory is sufficient.

Core Concepts of PagedAttention

To solve this, PagedAttention treats the GPU KV cache exactly like an operating system treats physical RAM.

  1. Blocks: The KV cache is divided into fixed-size "blocks". Each block can hold the attention keys and values for a specific number of tokens (e.g., 16 tokens).
  2. Logical vs Physical: A sequence perceives its memory as a contiguous stream of "logical blocks". Under the hood, these map to randomly scattered "physical blocks" in GPU memory.
  3. Block Table: A map that translates logical block indices to physical block addresses in real-time during the attention computation.

Because blocks don't need to be contiguous, external fragmentation drops to 0%. Internal fragmentation only happens in the very last block of a sequence (averaging half a block of waste, which is negligible).

Hands-On Example: Processing Two Requests

Let's trace how a KV Cache Allocator processes tokens in real-time. We configure our system with a block size of 4 tokens per block.

Step 1: The Prompts Arrive

  • Request 1: "Who is the CEO of" (5 tokens). Logical Blocks needed: 2 (Block 0 holds 4 tokens, Block 1 holds 1 token).
  • Request 2: "Write a poem" (3 tokens). Logical Blocks needed: 1 (Block 0 holds 3 tokens).

The BlockAllocator pulls free physical blocks from memory and assigns them via the Block Table.

Step 2: Autoregressive Generation

As the model generates tokens, they are written to the last block of their respective sequence.

  • Request 2 generates "about" (Token 4). It is placed in Phys 5. Phys 5 is now FULL (4 tokens).
  • Request 2 generates "space" (Token 5). The BlockAllocator kicks in. It pulls a new free physical block (e.g., Phys 1), maps it to Logical Block 1 for Request 2, and writes the token there.

Modeling the Block Allocator in Python

To make this concrete, here is a simplified Python model of how the system manages this dynamic allocation on the CPU side before issuing GPU kernels.

class BlockAllocator: def __init__(self, total_vram, block_size_bytes): # Initialize pool of free physical block indices num_blocks = total_vram // block_size_bytes self.free_blocks = list(range(num_blocks)) self.tokens_per_block = 16 def allocate_block(self): if not self.free_blocks: raise MemoryError("KV Cache is completely full!") return self.free_blocks.pop(0) def free_block(self, physical_idx): self.free_blocks.append(physical_idx) class Sequence: def __init__(self, prompt_tokens, allocator): self.logical_to_physical = [] # The Block Table self.num_tokens = 0 self.allocator = allocator # Process initial prompt for token in prompt_tokens: self.append_token(token) def append_token(self, token): # Check if we need a new block if self.num_tokens % self.allocator.tokens_per_block == 0: new_phys_block = self.allocator.allocate_block() self.logical_to_physical.append(new_phys_block) # Logically append the token self.num_tokens += 1 # In reality, a kernel is launched here to write the KV tensors # to GPU memory at physical address: new_phys_block * block_size

The Superpower: Memory Sharing & Copy-on-Write

Once you have a page table, you unlock the ability for multiple logical blocks to point to the exact same physical block. This is identical to memory sharing in operating systems.

In scenarios like Beam Search, Parallel Decoding, or handling multiple users with the same large system prompt, you can share the KV cache for the prompt across all requests. The physical memory stores the prompt once. The block table for Sequence A and Sequence B both point to those same physical blocks.

They only diverge when the generated output differs. At that moment, a "Copy-on-Write" mechanism allocates a new physical block for the diverging sequence.

Conclusion

By treating the LLM KV cache like an operating system treats RAM, PagedAttention enables models to serve significantly larger batches of requests concurrently. It practically eliminates memory waste and turns complex sampling methods (like beam search) from memory-hogs into highly efficient operations.
