Paged Attention from First Principles: A View Inside vLLM

Agent System Patterns · Deep read

SP
Sushant Pargaonkar June 2026 · 16 min read

Large language models are trained in highly parallel, compute-bound workloads — but serving them is a completely different beast. Inference is memory-bound and sequential. A key bottleneck is the key-value (KV) cache, which stores contextual information during decoding. Prior systems wasted 60–80 % of GPU memory to fragmentation, crippling throughput. PagedAttention, introduced in the vLLM paper by Kwon et al., fixes this by borrowing virtual memory paging directly from operating systems — achieving near-zero waste and 2–3 × higher throughput.

LLM Training vs Inference

To understand the memory crisis in serving, we first need to understand how inference differs from training. At the heart of all modern LLMs is the transformer architecture. The flow is always the same: raw text is tokenised, token ids are embedded into dense vectors, positional encodings are added, and then the sequence flows through stacks of decoder blocks. Each block applies causal self-attention followed by a feedforward sublayer, residual connections, and normalisation.

During training, the entire target sequence is known in advance. The model processes all N tokens in one forward pass; the causal mask ensures token t cannot attend to t+1, preserving the autoregressive property while still computing all positions in parallel. GPUs thrive here — their thousands of cores stay fully saturated with large matrix multiplications. Training is compute-bound. 1

During inference, the future tokens are unknown. The model must generate them one by one, appending each new token to the context and running another forward pass. Instead of large dense matmuls over many tokens, the GPU repeatedly loads the full model weights plus a growing cache for every single step. The bottleneck shifts from compute to memory bandwidth. This is why inference is memory-bound.

The Two Phases of Inference

Every LLM inference request goes through exactly two phases:

  • Prefill (prompt phase) — the model reads the entire input prompt in one forward pass and generates the very first output token. Because all N prompt tokens are processed together, this is large-matmul territory and remains somewhat compute-bound.
  • Decoding — the iterative loop. The model appends the last generated token to the context and runs another forward pass to produce the next token. This repeats until an end-of-sequence token or length limit is hit. Each step involves very few new tokens (usually just one), so the GPU spends most of its time loading data, not computing — fully memory-bound. 2

The KV Cache: What It Is and Why It Matters

At each transformer layer, the attention mechanism computes three matrices from the input: Query (Q), Key (K), and Value (V). The attention output is softmax(Q·Kᵀ/√d) · V. Because generation is autoregressive — every new token needs to attend to every previous token — you would normally need to recompute K and V for every previous token at each step. That's quadratically expensive.

The KV cache avoids this. After computing K and V for a token, you cache them. On the next step you only compute Q for the new token, then do a single attention operation against the cached Ks and Vs. This trades compute for memory: you pay once to compute K and V, then simply store them and read them back every decode step. 3

How large is the KV cache? For a model like LLaMA-2 13B with 40 layers, 40 heads, a head dimension of 128, float16 precision, and a context length of 4096 tokens: 2 × 40 × 40 × 128 × 4096 × 2 bytes ≈ 3.2 GB per request. A single A100 (80 GB) can therefore hold around 25 fully-saturated concurrent requests before running out of memory — if there's no fragmentation.

The Memory Fragmentation Crisis

Before PagedAttention, LLM serving systems allocated the KV cache for a request by reserving a contiguous block of GPU memory sized for the maximum possible output length (e.g., 2048 tokens). This was the only safe approach — without knowing the output length in advance, you had to reserve worst-case space up front or risk running out mid-generation.

This led to catastrophic memory waste in two forms. Internal fragmentation: if a request finishes at 150 tokens but had 2048 reserved, 1898 tokens' worth of memory sat idle and locked — unavailable to anyone else — until that request completed. External fragmentation: after several requests finish and release their blocks, the freed memory is scattered in small gaps that cannot accommodate a new large request, even if total free memory is theoretically sufficient. 4

The OS Analogy: Virtual Memory and Page Tables

Operating systems solved an identical problem decades ago. Early OSes gave each process a contiguous block of physical RAM. This caused the same fragmentation disaster — some processes used only a fraction of their allocation, others couldn't be loaded at all. The solution was virtual memory: decouple what a process sees (a flat address space) from where the data actually lives in physical RAM.

Physical RAM is divided into fixed-size pages (typically 4 KB). A process's virtual address space is divided into identically-sized virtual pages. The OS maintains a page table that maps virtual page numbers to physical frame numbers. The process sees a contiguous address space; the OS scatters the actual data wherever free physical frames exist. 5

PagedAttention applies this exact idea to the GPU KV cache. The KV cache is divided into fixed-size physical blocks. Each sequence perceives its KV cache as a contiguous stream of logical blocks. A block table maps logical block indices to physical block addresses. The GPU attention kernel uses this table at runtime to find where each K and V tensor actually lives.

PagedAttention: The Mechanics

In PagedAttention, each physical block holds the K and V tensors for a fixed number of tokens — the block size. A block size of 16 means each physical block stores 16 token positions worth of keys and values across all attention heads and layers. Blocks do not need to be contiguous in GPU memory.

When a new request arrives, the BlockAllocator reserves no memory upfront. It only allocates a new physical block when the current last block fills up. If a request finishes at token 37 with a block size of 16, it uses exactly 3 blocks (0–15, 16–31, 32–36) — the 11 unused slots in the last block are the only wasted space. External fragmentation drops to zero. Internal fragmentation averages half a block per request — negligible. 6

Hands-On: Modelling the Block Allocator

To make the mechanics concrete, here is a minimal Python model of how vLLM manages this on the CPU side before dispatching GPU kernels. The real vLLM implementation lives in vllm/core/block_manager.py and is significantly more sophisticated, but the core idea is exactly this.

from collections import deque from dataclasses import dataclass, field from typing import Dict, List # ───────────────────────────────────────────────────── # BlockAllocator: manages the pool of physical KV blocks # ───────────────────────────────────────────────────── class BlockAllocator: def __init__(self, total_gpu_blocks: int, block_size: int): # block_size = number of token positions each block holds self.block_size = block_size # free_blocks is a pool of physical block indices self.free_blocks: deque = deque(range(total_gpu_blocks)) self.ref_counts: Dict[int, int] = {} def allocate(self) -> int: if not self.free_blocks: raise RuntimeError("OOM: no free KV blocks") phys_id = self.free_blocks.popleft() self.ref_counts[phys_id] = 1 return phys_id def free(self, phys_id: int): self.ref_counts[phys_id] -= 1 if self.ref_counts[phys_id] == 0: self.free_blocks.append(phys_id) def fork(self, phys_id: int): # copy-on-write: bump ref count instead of copying data self.ref_counts[phys_id] += 1 # ───────────────────────────────────────────────────── # Sequence: owns a logical block table # ───────────────────────────────────────────────────── @dataclass class Sequence: seq_id: int # logical_to_physical[i] = physical block index for logical block i logical_to_physical: List[int] = field(default_factory=list) num_tokens: int = 0 def append_token(self, allocator: BlockAllocator): if self.num_tokens % allocator.block_size == 0: # current last block is full (or this is the first token) # allocate a brand new physical block new_block = allocator.allocate() self.logical_to_physical.append(new_block) self.num_tokens += 1 def physical_block_for_token(self, token_idx: int, block_size: int) -> int: logical_blk = token_idx // block_size return self.logical_to_physical[logical_blk] # ───────────────────────────────────────────────────── # Demo run # ───────────────────────────────────────────────────── allocator = BlockAllocator(total_gpu_blocks=8, block_size=4) seq = Sequence(seq_id=1) # Simulate processing "Who is the CEO of OpenAI" (6 prompt tokens) for _ in range(6): seq.append_token(allocator) print(f"After 6 tokens — blocks used: {seq.logical_to_physical}") # → [0, 1] (block 0: tok 0-3, block 1: tok 4-5 partial) # Decode: add 3 more tokens for _ in range(3): seq.append_token(allocator) print(f"After 9 tokens — blocks used: {seq.logical_to_physical}") # → [0, 1, 2] (block 2 allocated when tok 8 arrived) print(f"Free blocks remaining: {list(allocator.free_blocks)}") # → [3, 4, 5, 6, 7]

Notice what never happens: no 2048-token reservation. No upfront commitment. The allocator hands out exactly one new block the moment the previous one fills up. When the sequence finishes and is freed, all its physical blocks immediately return to the pool — ready for the next request.

Memory Sharing and Copy-on-Write

Virtual memory's page table abstraction unlocks another powerful capability: multiple sequences can point to the same physical block. This is directly analogous to shared memory pages between OS processes.

This matters enormously in practice. Beam search maintains K candidate sequences (beams) that all diverge from the same prompt prefix. Parallel sampling generates N independent outputs from the same prompt. Multi-turn chat often re-processes the same system prompt for every user. In all of these, the prompt's KV cache is identical across all sequences — why store it N times? 7

With PagedAttention's block table, you simply set multiple sequences' block tables to point to the same physical block for shared prefixes, and increment the block's reference count. No data is duplicated. When sequences diverge — generating different tokens — a Copy-on-Write (CoW) fires: the diverging sequence gets a new private physical block, the shared data is copied there once, and only that sequence's block table is updated to point to the new block. All others continue sharing the original.

The Attention Kernel: Under the Hood

The Block Table isn't just a bookkeeping structure — it is actively used inside the CUDA attention kernel at inference time. The modified attention kernel receives the block table alongside the query tensor and iterates through logical blocks to reconstruct the full K,V context.

# Pseudocode of the PagedAttention kernel inner loop # (real impl is CUDA C++ — this captures the logic) def paged_attention( query, # [num_heads, head_dim] — single query token block_table, # [num_logical_blocks] — maps log → phys kv_cache, # [num_phys_blocks, 2, block_size, num_heads, head_dim] context_len: int, block_size: int, ): attn_scores = [] for token_idx in range(context_len): logical_block = token_idx // block_size token_in_block = token_idx % block_size # ← Block Table lookup (this is the key indirection) phys_block = block_table[logical_block] k = kv_cache[phys_block, 0, token_in_block] # key v = kv_cache[phys_block, 1, token_in_block] # value score = dot(query, k) / sqrt(head_dim) attn_scores.append((score, v)) weights = softmax([s for s, _ in attn_scores]) return sum(w * v for (_, v), w in zip(attn_scores, weights))

The one-line indirection — phys_block = block_table[logical_block] — is the entirety of PagedAttention's magic at the kernel level. The rest of the attention computation is unchanged.

Conclusion

PagedAttention is an elegant case of systems thinking applied to deep learning infrastructure. The idea is conceptually simple — page tables have existed in OS kernels since the 1960s — but applying it to the KV cache required rethinking how LLM serving engines allocate, track, and free GPU memory at runtime, and building CUDA kernels that can efficiently perform scattered, indirect memory reads.

The results speak for themselves: near-zero fragmentation, 2–3× throughput improvement, and radical efficiency gains for beam search and shared-prefix workloads. PagedAttention is now the standard memory management strategy in vLLM, TensorRT-LLM, and HuggingFace TGI. If you're serving LLMs at any scale, you're almost certainly benefiting from it already.

The one-line summary: Instead of reserving a contiguous slab of GPU memory per request (and wasting most of it), PagedAttention allocates memory in small fixed-size blocks on demand, using a block table to translate logical positions to physical locations — exactly like an OS page table maps virtual to physical addresses.
1

Compute-bound vs Memory-bound

During inference, the GPU spends most of its time loading weights and KV cache from HBM (High Bandwidth Memory), not doing arithmetic. This is why memory bandwidth — not FLOPS — is the key metric for serving performance.

2

Prefill vs Decode latency

Prefill is fast despite processing N tokens because the GPU runs them in parallel. Decoding is slower per token and dominates wall-clock time for long outputs — any memory saving here multiplies across every token generated.

3

KV Cache size formula

For LLaMA-2 13B (40 layers, 40 heads, dim 128, bfloat16) at 4096 context: ≈ 3.2 GB per request. A single A100 80 GB holds only ~25 requests if memory is perfectly managed — far fewer with fragmentation.

4

Fragmentation types visualised

Internal frag: memory reserved but never written. External frag: holes between allocations too small for a new request. PagedAttention eliminates external fragmentation entirely and reduces internal to half a block per sequence.

5

OS virtual memory page table

Exactly the same idea: the process sees a clean flat address space. The page table (maintained by the OS kernel) silently maps it to wherever the physical frames actually live. PagedAttention makes vLLM the "OS kernel" for GPU KV memory.

6

Block size trade-off

vLLM defaults to block_size = 16, which balances internal fragmentation (average 8 tokens wasted per sequence) against kernel efficiency and block table size. You can tune this in the vLLM config.

7

Beam search memory saving

In beam search with K=4, the original approach stores the prompt KV cache K times. With PagedAttention's reference counting, the prompt is stored once. Reported savings in the paper: 55 % less memory for beam search workloads.
