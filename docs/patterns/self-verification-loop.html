Self-Verification Loop

Make the agent prove the behavior, not just narrate completion.

This pattern adds a second loop around agent output: build, verify behavior, inspect failure evidence, repair, and rerun. A UI or API is not accepted because it looks plausible. It is accepted only when independent checks confirm the intended state transition happened.

Source signal

Replit's automated self-testing work points at a simple truth: an agent needs behavioral evidence, not just a completion claim.

Repo addition

Core claim

Generated output can be convincing while still broken. The system needs a verifier that executes user intent and captures durable evidence.

Verifier questions

  • Can a user complete the intended action?
  • Did state change in the right place?
  • Did the check observe behavior, not just structure?

Best use cases

  • User-facing flows and tool-driven tasks.
  • Cases where screenshots or file existence are misleading.
  • Workflows where failure evidence can guide repair.

Implementation shape

The Python model includes an application, executable scenarios, verification steps, evidence capture, and a repair loop that reruns the same behavior after fixes.

Logic

What separates strong verification from shallow checks.

1. Execute intent

The verifier should interact with the artifact as a user or caller would, not merely inspect whether elements exist.

2. Capture evidence

Record state changes, logs, outputs, and failure assertions so the repair step has something concrete to reason from.

3. Repair from failure

The repair step should target the observed gap, not wander into generic improvement mode.

4. Rerun the same scenario

The system is only done when the original failing behavior passes under the same executable check.

Verification

What the experiments show.

Shallow false pass

A success-looking signup flow passes a naive checker even though no persistent state actually changes.

Behavioral catch

The stronger verifier executes the flow, inspects durable state, and rejects the Potemkin app.

Repair rerun

After a targeted fix, the system reruns the same scenario and accepts the app only when the state mutation is real.

Related Controls

What to pair this with.

Before verification Decision-Time Guidance

Use this when repeated failures show the agent needs a targeted nudge before another attempt.

 After verification Reversible Sandbox Execution

Use this when passing behavior is not enough; the diff still needs promotion policy.
