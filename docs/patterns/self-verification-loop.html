Accept behavior only after proof.

Agent System Patterns · Note 02

SP
Sushant Pargaonkar June 2026 · 5 min read

This note adds a second loop around agent output: build, verify the behavior, inspect the failure evidence, repair, and rerun. A UI or API isn't accepted because it looks plausible — only when independent checks confirm the intended state change actually happened.

The core idea

Generated output can be convincing while still broken. The system needs a verifier that executes the user's intent and captures durable evidence — not one that merely checks whether elements exist on the page.

The questions a verifier asks

  • Can a user complete the intended action?
  • Did state change in the right place?
  • Did the check observe behavior, not just structure?

When to reach for it

  • User-facing flows and tool-driven tasks.
  • Cases where screenshots or file existence are misleading.
  • Workflows where failure evidence can guide the repair.

How it works

  1. Execute the intent. Interact with the artifact as a user or caller would, not by inspecting whether elements exist.
  2. Capture evidence. Record state changes, logs, outputs, and failed assertions so the repair step has something concrete to work from.
  3. Repair from failure. Target the observed gap — don't wander into generic improvement mode.
  4. Rerun the same scenario. The work is only done when the original failing behavior passes under the same executable check.

What the experiments show

  • A success-looking signup flow passes a naive checker even though no persistent state changes.
  • A stronger verifier executes the flow, inspects durable state, and rejects the Potemkin app.
  • After a targeted fix, the system reruns the same scenario and accepts the app only when the state mutation is real.

How it's built

The implementation includes a sample application, executable scenarios, verification steps, evidence capture, and a repair loop that reruns the same behavior after fixes.
