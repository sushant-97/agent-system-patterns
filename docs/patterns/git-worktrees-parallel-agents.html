One repo. Many agents. Zero conflicts.

Agent System Patterns · Engineering

Sushant Pargaonkar June 2026 · 11 min read

You have three AI agents. Each one is good at its job. The moment you point all three at the same working directory, they immediately start wrecking each other's work — overwriting files, reading half-written state, producing diffs that conflict before you've reviewed either. Git worktrees are the missing piece: a way to give every agent its own isolated desk inside the same building, without duplicating the building.

The problem with one working directory

When multiple agents share a single directory, three things go wrong. The first is lost writes — Agent A reads auth/middleware.ts, thinks for a moment, then writes its changes back. Agent B read the same file slightly earlier and overwrites A's version without ever seeing A's edits. Neither agent knows this happened; both report success.

The second is corrupted context. An agent's reasoning about a file is built on what that file said when the agent read it. If another agent modifies it mid-task, the first agent is now reasononing about a file that no longer exists in the form it was read. The conclusions it draws are about a ghost.

The third is lock contention. Git itself uses lock files — .git/index.lock, .git/COMMIT_EDITMSG — to protect its own state. Two agents both trying to stage files at the same moment will see errors. One agent loses its commit. If you're not watching, it silently disappears.

The instinct is to reach for separate clones. Clone the repo three times, give each agent a clone. This works but wastes everything: disk, fetch time, CI setup. A repo with 30,000 files takes roughly 1 GB per clone, and a fresh clone averages 90 seconds. With ten agents you'd clone for 15 minutes and use 10 GB before any agent writes a single line. There's a better primitive built into git itself.

What git worktrees actually are

A git repository and a working directory are not the same thing. The repository is the .git/ folder — the object store, the ref database, the config. The working directory is just a view into that data: checked-out files you can read and edit. One repository can serve any number of working directories simultaneously. That's a worktree.

Each linked worktree gets its own HEAD (its own current branch), its own index (its own staging area), and its own working files on disk. None of that is shared. What is shared is everything inside .git/objects/ — commits, trees, blobs, the entire history. This is why a worktree checkout takes about 20 seconds instead of 90: it reads from a local object store that's already fully populated. It only writes out the working files.

The mechanism is a small indirection. Inside each linked worktree, the expected .git/ directory is replaced with a plain text file named .git containing a single line: gitdir: /path/to/repo/.git/worktrees/feature-auth. Git follows that pointer to find its data. The linked worktree's private state (HEAD, index, reflog) lives in .git/worktrees/feature-auth/ in the main repo. Everything else — all objects, all refs under refs/ — is shared from the single object store in the main repo's .git/.

The commands you actually need

The git worktree API is small. Creating a worktree requires two things: a path for the new working directory, and a branch to check out there. If the branch doesn't exist yet, -b creates it:

# Create a new worktree on a new branch git worktree add ../my-repo-auth -b feature/auth # Create a worktree for an existing branch git worktree add ../my-repo-hotfix hotfix/null-check # See all active worktrees git worktree list # Remove a worktree (also deletes the directory) git worktree remove ../my-repo-auth # Clean up stale worktree registrations (after manual dir deletion) git worktree prune

By convention, many teams keep all worktrees as siblings of the main checkout, with a shared parent directory. A common layout looks like this:

~/projects/ my-repo/ # main checkout (.git/ lives here) my-repo-auth/ # worktree: feature/auth my-repo-payments/ # worktree: feature/payments my-repo-tests/ # worktree: chore/test-suite

A thin shell function makes the pattern easy to repeat. This one creates a worktree, copies the .env file, installs dependencies, and opens the directory:

function wt-new() { local name="$1" local branch="feature/$name" local dir="../$(basename $(pwd))-$name" git worktree add "$dir" -b "$branch" # Copy environment config cp .env "$dir/.env" 2>/dev/null || true # Install deps in the new tree (cd "$dir" && npm install --silent) echo "Worktree ready at $dir on branch $branch" } # Usage: wt-new auth # → ../my-repo-auth on feature/auth wt-new payments # → ../my-repo-payments on feature/payments

The parallel agent pattern

Once worktrees exist, the canonical pattern for parallel agent execution follows three phases: decompose, dispatch, and promote.

Decompose. An orchestrator (which may itself be an agent, a script, or a human) breaks the overall task into units that touch non-overlapping parts of the codebase. The decomposition rule is strict: if two units would modify the same file, they must be executed sequentially — not in parallel. Units that touch completely separate modules, directories, or concerns can run in parallel.

Dispatch. The orchestrator provisions one worktree per unit, starts one agent per worktree, and passes each agent its task and the path to its working directory. The agents work independently. Agent A never reads from Agent B's directory; it only reads from its own. All shared history is still accessible through the common object store — agents can inspect old commits, read tags, check history — but they never touch each other's in-progress files.

Promote. When each agent completes, the orchestrator reviews its diff — git diff main..feature/auth — and decides whether to merge, rebase, or discard. Because each worktree is a regular git branch, merging is just a merge. There's no special protocol. The branch from one worktree is visible in every other worktree immediately, because they share the same object store.

Claude Code's native worktree support

Claude Code added built-in worktree management in 2025. The --worktree flag (also -w) does the full setup in one command: creates a worktree at .claude/worktrees/<name>, checks out a new branch named worktree-<name>, and starts the Claude session with its working directory pointed at that path:

# Start a Claude session in an isolated worktree claude --worktree feature-auth # Two agents, two terminals, zero conflicts claude --worktree feature-auth # terminal 1 claude --worktree bugfix-payments # terminal 2 # Auto-named worktree (UUID suffix) claude --worktree

For automated pipelines where an orchestrator Claude spawns subagents, each subagent can request its own worktree via a frontmatter directive. The orchestrator provisions a fresh worktree before the subagent starts and auto-cleans it when done:

# In a subagent definition file: --- name: auth-agent description: Implement authentication module isolation: worktree # ← provisions a fresh worktree automatically --- Implement the OAuth2 flow in src/auth/. Add refresh token support. Follow the existing patterns in src/users/.

Cleanup is automatic for subagent worktrees that have no uncommitted changes, no untracked files, and no unpushed commits — they're swept after cleanupPeriodDays (default: 7). Worktrees you create directly with --worktree are never auto-removed, since the assumption is you want to review the branch before deciding.

What worktrees don't solve

Worktrees are a file-system isolation primitive. They're excellent at what they do, and they're not the whole picture. A common misconception is that worktrees eliminate all parallelism problems. They eliminate all file-level conflicts. Runtime conflicts remain.

Port collisions. Two agents that each spin up a development server both try to bind port 3000 by default. The second agent fails at startup or silently connects to the first agent's server. The fix is to assign each worktree a port offset based on its index: Agent 0 uses ports 3000–3009, Agent 1 uses 3010–3019, and so on. Set this in the environment before starting each agent:

# Derive port offset from worktree index export PORT=$((3000 + WORKTREE_INDEX * 10)) export API_PORT=$((4000 + WORKTREE_INDEX * 10)) # Pass to the agent environment WORKTREE_INDEX=1 claude --worktree feature-auth

Database state. Two agents seeding or migrating the same test database will corrupt each other's data. The cleanest solution at the integration-test level is per-worktree SQLite databases (file path includes the worktree name). For shared PostgreSQL or MySQL instances, prefix test schemas or databases with the worktree name: test_auth_wt1, test_payments_wt2. In CI, containerized databases per agent are cleaner.

Shared caches and secrets. Package managers cache globally by default. Two agents running npm install simultaneously on the same machine can corrupt the npm cache. Point each worktree at its own cache directory, or accept the race and retry. Secrets (.env files) need to be copied into each worktree directory at setup time — they're not shared by default, which is a feature: each worktree gets its own copy you can modify for per-task configuration.

For heavy parallelism — ten or more concurrent agents — the reliable answer is container isolation: each agent in its own Docker namespace with its own network stack, its own database container, its own port space. Git worktrees can still be used inside the shared object store mounted into each container; the isolation layers compose.

Task decomposition that actually works

The decomposition step is where most parallel agent setups fail. People underestimate how often "separate" features share the same files. The rule is simple but requires honest analysis: if two tasks would touch the same file, they must run sequentially, not in parallel.

Good parallel decomposition looks like domain separation. Authentication and payment processing live in different directories and rarely share files beyond the user model. A test suite update and a documentation pass are completely disjoint. Frontend component work and backend API changes touch different layers. These are safe to parallelize.

Bad parallel decomposition looks like feature-plus-refactor. "Agent A adds the new feature while Agent B refactors the shared utilities" always collides. Agent A is building on the utilities as they exist now; Agent B is removing or renaming them. When both branches try to merge, one of them will have built on a foundation that no longer exists.

A practical pattern for larger teams is a shared task document — a markdown file committed to the repo — that the orchestrator writes before agents start. Each entry has a task, a status field (queued / in-progress / done), and a list of files it will touch. An agent picks a task by atomically updating its status, then only reads and writes the files listed for its task. This makes the dependency graph explicit and catchable: before any task runs, a simple overlap check across the file lists finds conflicts before they happen.

Merging and reviewing agent output

Because each worktree is a regular branch, reviewing what an agent produced is just a normal diff review. Before merging anything:

# Review everything an agent changed relative to main git diff main..feature/auth # Or see just the file list git diff --name-only main..feature/auth # Check the commits the agent made git log main..feature/auth --oneline # Merge when satisfied git checkout main git merge feature/auth --no-ff # Or rebase to keep history linear git rebase main feature/auth git checkout main git merge feature/auth --ff-only

The --no-ff flag creates a merge commit that marks where the agent's work entered main — useful for auditing. The rebase approach keeps a linear history at the cost of rewriting the agent's commit hashes, which is fine for agent-produced work since you're the reviewer, not a collaborator expecting stable refs.

If two agent branches both need to merge and they touch adjacent (but not identical) code, you may hit a merge conflict. This is usually a signal that the decomposition was too tight — the tasks were closer together than the file list suggested. Resolve it manually, or feed the conflict back to an agent with explicit context about what both branches were trying to do.

The performance case

The concrete numbers make the argument for worktrees over clones. For a representative JavaScript monorepo with 30,000 files:

  • Clone: ~90 seconds, ~1 GB disk per copy. Five agents = 7.5 minutes of setup, 5 GB disk.
  • Worktree: ~20 seconds, ~150 MB disk per copy. Five agents = 1.5 minutes of setup, 750 MB disk.

For a large Go monorepo with ~200 MB of git objects, five clones would consume 1 GB of history storage. Five worktrees share the same 200 MB. The object store is written once and read by everyone.

Beyond setup time and disk, the performance argument is about wall-clock time on real tasks. A 2025 case study running four parallel agents on a feature decomposed into four domain-separated worktrees completed work that would have taken 30 hours sequentially in 8.4 hours. The agents worked at full speed with no coordination overhead because they literally never saw each other's changes until the merge step.

The decision you have to make first

Git worktrees are not the thing that makes parallel agents useful. Good task decomposition is the thing that makes parallel agents useful. Worktrees are what makes it safe to actually run them. If you skip the decomposition analysis and hand two agents overlapping tasks, worktrees will ensure their partial results don't corrupt each other on disk — but you'll still end up with two branches that can't be merged without human intervention.

The right mental model: worktrees are isolation infrastructure, not a coordination strategy. They buy you the ability to run things in parallel safely. What you run, in what order, with what dependencies between tasks — that's still on you to design. Do it well and the speedup is real. Skip it and you've just added complexity to a sequential process.
